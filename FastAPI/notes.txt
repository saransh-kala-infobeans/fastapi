# API Authentication — Concepts

## HTTP Headers in Auth
- After login, the server gives the client a token.
- For every request after login, the client sends this token in the request header:
  ```
  Authorization: Bearer <token>
  ```
- `Authorization` → header name
- `Bearer` → means "I am carrying a token"
- The long string after Bearer → the actual JWT token
- The server reads this header and validates the token before the handler function even runs.

---

## JWT — JSON Web Token
- A JWT is a string the server generates after a successful login.
- The client carries it in every subsequent request.
- Structure — 3 parts separated by dots:
  ```
  header.payload.signature
  ```

### Header
- Metadata about the token.
- Contains the algorithm used to sign, and the token type.
  ```json
  {"alg": "HS256", "typ": "JWT"}
  ```

### Payload
- The actual data/claims stored in the token.
  ```json
  {"userID": 1, "username": "john", "exp": 1719000000}
  ```
- `exp` → expiry timestamp, after which the token is invalid.
- JWTs are **not encrypted, just encoded (Base64URL)** — anyone can decode and read the payload. So never store passwords or sensitive info in it.

### Signature
- A tamper-proof seal on the token.
- The server creates it by running a calculation using the header, payload, and a secret key:
  ```
  header + payload + SECRET_KEY  →  calculation  →  signature
  ```
- When the client sends the token back, the server redoes the same calculation and compares the result with the signature in the token.
  - Match → valid ✅
  - No match → tampered, rejected ❌
- A hacker can change the payload, but can't fix the signature without knowing the secret key.

---

## Secret Key
- Just a string that lives on the server:
  ```python
  SECRET_KEY = "mysecretkey123"
  ```
- **Same for all users** — it's a server-level secret, not per-user.
- Never shared with anyone — only the server knows it.
- What changes per user is the **payload**, not the key.

---

## How a JWT is Created — Full Flow
1. User sends credentials (username + password) to the server.
2. Server verifies — does the user exist? Is the password correct?
3. If yes, the server itself creates the header and payload.
4. Server signs them using the secret key → produces the signature.
5. Combines all three → sends the JWT token back to the client.
6. Client stores the token and sends it in the `Authorization` header for every future request.
7. Server validates the token on every request before allowing access.

next, Status codes  401 and 403

1) Status code 401 - UNAUTHORIZED
--"don't know who you are".
-means, 
    the request is missing a Token, or the token is expired.
No token      -> 401
Invalid token -> 401
Expired token -> 401

How can a token expire??
Answer:
    Remember, the payload contains an `exp` field:

    payload:
    {"userID": 1, "username": "john", 
    "exp": 1719000000}

    That `exp` is just a **timestamp** — a specific point in time, after which the token is considered expired.

    The server, when creating the token, sets this intentionally:
    #server says: this token is only valid for 30 minutes from now
    exp = current_time + 30 minutes

    When the client sends the token back, the server checks two things:
    1. Is the signature valid?       → not tampered?
    2. Is current_time < exp?        → not expired?

    If the current time has passed the `exp` value → token is expired → `401`.

    ## Why Do Tokens Expire?

    Security. If a token gets stolen, it shouldn't be valid forever. Expiry limits the damage window.

    When a token expires, the user just logs in again and gets a fresh token.


2) Status code 403 - FORBIDDEN
--"I know you, but you're not allowed to do this"
--means, 
    The token is valid, but you don't have the permission for what you're trying to do.

Depends() and Dependency Injection

what's Dependency Injection?
It's just the idea of-- 
    "Before calling this function, run that function, and pass its result in"
That's it. Simple.

It's use?
    Without this, you'll have to manually write the whole checking process in EACH handler function. 
    which includes, 
    -extracting token from header
    -validate token.
    -get user from token.
    -then finally do the actual work.

With Dependency Injection
what you can simple do Is

@app.get("/task")
def get_tasks(user = Depends(get_current_user)):
    ...
    ...

#inside Depends() you write the function name, which you want to first run.
this way, all you have to simply do is to write a separate function the handles auth, for ONE time.

def get_current_user(token: str = Header()):
    .... 
    perform validation etc.
    return user

Request comes in
      ↓
FastAPI sees Depends(get_current_user)
      ↓
Runs get_current_user first
      ↓
Token invalid → raises 401 → request dies here
Token valid   → returns user → passed into handler
      ↓
Handler function runs normally

-> Depends() is just a way of saying "this route has a prerequisite — run this function first."
